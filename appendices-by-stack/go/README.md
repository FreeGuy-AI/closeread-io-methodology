# Go appendix

Tools an auditor reaches for when the primary stack is Go, the typical false positives to suppress, and the minimum acceptable depth of coverage for a Closeread-grade packet on a Go codebase.

"Go" in audit terms covers single-binary services (the dominant case in acquisition-side diligence), CLI tools, libraries published as importable modules, and the increasingly common "monorepo with `cmd/` per binary" pattern. Module resolution is almost always `go.mod` + `go.sum` against the public module proxy at `proxy.golang.org`, with `GOSUMDB=sum.golang.org` for cryptographic verification. Vendored dependencies under `vendor/` exist but are now a minority pattern. The audit posture changes meaningfully across these layers, but the toolchain below covers all of them.

## Tools

| Zone | Tool | Notes |
|---|---|---|
| SCA | `govulncheck` (official), `nancy`, `osv-scanner` | `govulncheck` from the Go team is the canonical scanner and is reachability-aware: it walks the call graph from `main` and reports only the CVEs the binary actually reaches. `nancy` (Sonatype) reads `go list -m -json all` against the OSS Index. `osv-scanner` adds the OSV breadth. Run all three; cross-reference. |
| Credentials | `gitleaks`, `trufflehog` | Same as Node, Python, and Ruby. Watch for `embed.FS` directives that pull a `.env` or a credentials JSON into the compiled binary; the file does not appear in the source tree at runtime but the secret is shipped. |
| Third party | `go list -m -json all` + manifest review | `go.mod` lists direct deps; `go list -m -json all` enumerates the full transitive graph after MVS (Minimum Version Selection) resolves. `go.sum` is the cryptographic ledger. Parse all three against the vendor signature table. |
| Test coverage | `go test -cover`, `go test -coverprofile`, `gocov` | Built into the toolchain; no third-party runner required. `go test ./... -coverprofile=cover.out` followed by `go tool cover -func=cover.out` gives the per-function percentage. Check for a `Makefile` or `taskfile.yml` target that pins coverage flags. |
| Security | `gosec`, `staticcheck`, `semgrep --config=auto` | `gosec` is the canonical Go security linter (formerly under `securego/`). `staticcheck` catches correctness bugs that shade into security (unchecked errors that hide injection paths). Semgrep adds polyglot patterns. |
| License | `go-licenses` (Google), `wwhrd` | `go-licenses` from Google walks the module graph, fetches each license file, and classifies the SPDX identifier. `wwhrd` enforces an allowlist defined in `.wwhrd.yml`. The license metadata is not in `go.mod`; it lives in each module's source tree, which is why a network-aware tool is required. |
| Architecture | `go mod graph`, `goda`, `golangci-lint` import analyzers | `go mod graph` emits the module dependency graph as text. `goda` (Loov) gives a richer query language over the package graph (in-repo edges, not just module edges). The standard layout (`cmd/`, `pkg/`, `internal/`) is the project's first architectural statement; an empty or absent `internal/` in a project larger than ~10K LOC is itself a finding. |
| Dependency policy | Dependabot (`gomod` ecosystem), Renovate | Dependabot supports Go modules natively via `package-ecosystem: gomod` in `.github/dependabot.yml`. Renovate covers the same ground with finer control. Absence of either on a Go repo with a `go.mod` is a hireability finding, not just a security one. |

## Typical false positives

* `govulncheck` reports a vulnerability against a module that is in `go.sum` but is not actually imported by any reachable path from `main`. This is the tool working correctly: the dep was pulled in by a build-tag or test-only path and the production binary does not link it. A clean packet names the CVE, names the module, and quotes `govulncheck`'s own reachability verdict (`No vulnerabilities found in the binary`, or the specific call site if found). Do not promote a `govulncheck` finding without reading its `-json` output end to end.
* `nancy` flags advisories against modules that the Go toolchain itself depends on (e.g. `golang.org/x/crypto` minor versions) when the project pins a newer version transitively. Run `go list -m -versions <module>` to see the actual resolved version before promoting.
* `gosec` rule `G404` flags `math/rand` usage as a CSPRNG weakness even when the call is `rand.Intn(len(slice))` for shuffling display order in a CLI tool. Read the call site before promoting; a genuine finding is a `math/rand`-derived token, session ID, or password reset key, not a UI shuffle.
* `gosec` rule `G304` ("file path provided as taint input") fires on `os.ReadFile(filepath.Join(configDir, name))` even when `name` is a hard-coded constant. Audit the caller before promoting.
* `gosec` rule `G115` (integer overflow) fires aggressively on any `int` to `int32` conversion. On 64-bit hosts this is almost always a benign narrowing for a value with a known small range; verify the upper bound is bounded by application logic, then suppress with `#nosec G115 -- bounded by upstream validation` plus an inline comment.
* `go-licenses` cannot classify modules whose `LICENSE` file is missing, named non-standardly, or written in a non-English variant. The tool will print `Unknown` for these; the packet must resolve each `Unknown` by reading the upstream repository, not by suppressing.
* `gitleaks` flags Go test fixtures under `testdata/` that contain example API keys, fake JWTs, and bearer tokens. Mark `is_sensitive: false` and add a note that the value is under `testdata/` (the conventional directory the Go toolchain ignores during `go build`).
* Generated code under `*_gen.go`, `*.pb.go` (protobuf), and `mock_*.go` (mockgen, gomock) routinely fails `gosec` and `staticcheck` checks that would matter in hand-written code. Suppress with the `//go:generate` provenance plus the generator name, not globally.

## Minimum acceptable depth

* All eight tool zones above run cleanly with no skipped checks on the most recent commit. If the project pins a Go release that `govulncheck` or `staticcheck` no longer supports, the packet names the Go version, the EOL position (see below), and the upgrade work required before the scanners run.
* `govulncheck` is run with `-mode=binary` against the compiled artifact in addition to `-mode=source` against the module. The binary pass catches what the source pass misses when build tags or cgo affect which files compile in. Both outputs are quoted in the packet.
* `gosec` runs with default confidence and full ruleset; results capped at 8 high, 12 medium, 6 low in the packet to avoid finding-flood. Any suppressed rule is named with a one-line rationale and a `#nosec` annotation in source.
* Test coverage measured, not assumed: actual percentage from `go test ./... -coverprofile=cover.out` plus `go tool cover -func=cover.out`. Per-package coverage is quoted, not just the project total, because a Go project commonly has a 95% covered `pkg/` and a 3% covered `cmd/`. A blended number hides which side is which.
* The third-party vendor inventory cross-references the full transitive set from `go list -m -json all` (not just direct deps from `go.mod`) against the vendor signature table. Any module not in the table gets a manual lookup before the packet ships. Modules with no published license metadata in their source tree get flagged separately and resolved by reading the upstream repo.
* `go.mod` + `go.sum` verification is performed: `go mod verify` is run and its output (`all modules verified` or the specific tamper finding) is quoted. `go.sum` must be present and committed; its absence on a project with a `go.mod` is a hard finding because it disables the cryptographic chain to `sum.golang.org`.
* Vendored deps under `vendor/`, when present, are diffed against the module proxy version: `go mod vendor` is re-run in a clean checkout and the result is compared against the committed `vendor/` tree. A drift between the two is a finding (the vendored copy has been edited locally and no longer matches what `go.sum` would verify).
* Standard layout review names where the project sits: `cmd/<binary>/main.go` for each binary, `pkg/` for importable public code, `internal/` for code the compiler refuses to expose outside the module. A project larger than ~10K LOC with no `internal/` directory has consciously or unconsciously published its full surface as a public API, which is a hireability and IP-ownership signal worth surfacing.
* cgo surface is explicitly inventoried. Any package that imports `"C"` is named with its file path and a one-line description of what native library it binds. cgo is a risk surface because it bypasses Go's memory safety, defeats `govulncheck` reachability analysis (which stops at the cgo boundary), and introduces a non-Go build dependency that the hireability writeup must account for. A project that uses cgo for SQLite (`mattn/go-sqlite3`), image processing, or crypto bindings needs the native dep version named, not just the Go wrapper.
* Reflect-heavy patterns are surfaced separately from generics. The `reflect` package, runtime type assertions in hot paths, and `unsafe.Pointer` usage are each flagged with a file:line citation and a maintainability note. Generics (Go 1.18+) are noted where the project uses constrained type parameters; a project on Go 1.22+ that still uses `interface{}` (now `any`) for collection-of-mixed-types where generics would fit is a refactor signal, not a finding.
* Environment loading is audited. Go has no canonical `.env` story; projects use `joho/godotenv`, `spf13/viper`, `caarlos0/env`, or hand-rolled `os.Getenv` wrappers. The packet names which one and whether `.env*` files appear in `.gitignore`. A project that uses `godotenv.Load()` in `main` without a `_, _ = godotenv.Load()` swallow pattern, and that ships without `.env` in `.gitignore`, is a credential-exposure finding waiting to happen.
* Dependabot or Renovate configuration is verified. `.github/dependabot.yml` with `package-ecosystem: gomod` and a sensible interval (`weekly` minimum) is the baseline. Absence is a finding for any Go repo with more than ~5 direct deps; the SCA tools will catch the same CVEs after they ship, but the dependency-policy writeup needs to say whether the project is set up to learn about them at all.
* Go version EOL is named explicitly. Go follows a rolling-window support policy: the Go team supports the latest two minor releases and drops the third when a new one ships. As of 2026, Go 1.22 and 1.23 are supported; Go 1.21 is EOL. A project pinned in `go.mod` to a version older than the supported window (`go 1.20`, `go 1.21`, or earlier) gets the EOL flag plus the upgrade-cost writeup. The `toolchain` directive in `go.mod` (Go 1.21+) further pins the minimum toolchain version; mismatches between `go` and `toolchain` lines are worth a line in the packet.

### What deterministic tools cover, and what they do not

Deterministic Go tools (`govulncheck`, `nancy`, `gosec`, `staticcheck`, `go-licenses`, `go test -cover`, `go mod verify`) cover the SCA, SAST, coverage, license, and module-integrity zones cleanly. The Go toolchain is unusually well-built for audit work: reachability analysis is first-class, the module proxy provides a cryptographic chain by default, and `go vet` plus `staticcheck` catch a meaningful fraction of correctness bugs without configuration.

What they do not cover well, and what an AI specialist pass adds: reading a project's `internal/` decomposition to detect whether the boundary is load-bearing or cosmetic, distinguishing idiomatic context-aware code (`ctx context.Context` threaded through every call) from a project that ignores cancellation in long-running goroutines, recognizing whether a `cmd/` proliferation is a real multi-binary product or accidental copy-paste, and surfacing the architectural drift that happens when a Go service grows generics-heavy abstractions on top of an originally interface-first design. Those are the findings that a Closeread-grade Go packet adds on top of the deterministic baseline.
