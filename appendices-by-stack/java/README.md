# Java appendix

Tools an auditor reaches for when the primary stack is Java, the typical false positives to suppress, and the minimum acceptable depth of coverage for a Closeread-grade packet on a Java codebase.

"Java" in audit terms is two ecosystems running on the same runtime. The first is the build-tool split: Maven (XML-driven `pom.xml`, dominant in enterprise and bank-adjacent codebases) and Gradle (Groovy or Kotlin DSL `build.gradle` / `build.gradle.kts`, dominant in Android and newer JVM startups). The second is the framework split, where Spring Boot is the gravitational center for server-side Java and detection of its presence reshapes the entire audit posture. Kotlin/JVM is Java-adjacent for audit purposes; it consumes the same Maven and Gradle dep tree, runs on the same JVM, and inherits the same CVE surface, but its source-level tooling diverges (detekt, ktlint, Kotlin-specific Semgrep rules). A mixed Java + Kotlin module is normal in 2026 and the packet covers both source trees as one stack.

## Tools

| Zone | Tool | Notes |
|---|---|---|
| SCA | OWASP `dependency-check`, `snyk test`, `osv-scanner` | OWASP `dependency-check` is the canonical free OSS SCA for Java; runs against Maven and Gradle. Snyk adds commercial breadth and a richer transitive resolver, especially for Gradle. `osv-scanner` cross-references the OSV.dev database and catches Maven Central coordinates that the other two miss. Run all three on a non-trivial codebase. |
| Credentials | `gitleaks`, `trufflehog` | Same as Node, Python, and Ruby. Watch for `src/main/resources/application.properties` and `src/main/resources/application.yml` (Spring Boot's default config locations) and for `META-INF/secrets/` directories that some legacy projects use. Spring Boot's `spring.datasource.password=` lines are the most common live-credential-in-repo finding. |
| Dependency hygiene | GitHub Dependabot (Maven and Gradle ecosystems), Renovate | Dependabot supports `maven` and `gradle` ecosystems natively; the `dependabot.yml` config gates which directories get scanned. Renovate has finer-grained version-range control for Gradle's `libs.versions.toml` (Gradle version catalog) and for Maven property-driven version pins. A clean packet names which is configured, the update cadence, and any auto-merge rules. |
| Third party | manifest review + source grep | Parse `pom.xml`, all `build.gradle` / `build.gradle.kts` files, `libs.versions.toml` if present, and the resolved dependency tree (`mvn dependency:tree` or `gradle dependencies`) against the vendor signature table. The resolved tree is authoritative because Maven and Gradle both apply conflict-resolution rules that the manifest alone does not express. |
| Test coverage | JUnit 5 + JaCoCo, JUnit 4 (legacy), TestNG (niche) | JaCoCo is the dominant JVM coverage instrumenter and reports both line and branch coverage. The Maven plugin emits `target/site/jacoco/jacoco.xml`; the Gradle plugin emits `build/reports/jacoco/test/jacocoTestReport.xml`. JUnit 5 (Jupiter) is the default in any project started after roughly 2020; JUnit 4 lingers in older modules and the `@RunWith` annotation is the giveaway. TestNG appears mostly in QA-adjacent and Selenium-derived codebases. |
| Security | `spotbugs` + `find-sec-bugs` plugin, `semgrep --config=auto`, OWASP `dependency-check` (overlaps SCA) | SpotBugs with the find-sec-bugs plugin is the canonical free SAST for Java and catches the deserialization, JNDI, and XXE patterns that produced Log4Shell, Spring4Shell, and the Jackson RCE chain. Semgrep adds polyglot and modern-framework patterns SpotBugs lags on. Both should run. |
| Static quality | `checkstyle`, `pmd`, `errorprone`, `detekt` (Kotlin) | Not security tools per se, but `pmd` and `errorprone` surface the unsafe-pattern accumulations (unchecked casts, deprecated API use, race conditions in static fields) that map to long-term maintenance risk in a key-person finding. `detekt` is the Kotlin equivalent of `pmd` and is required when Kotlin source is present. |
| License | Maven `license-maven-plugin`, Gradle `license-gradle-plugin` (Hierynomus), `license_finder` | License metadata lives in each artifact's POM (or in `MANIFEST.MF` for older JARs). Pull the report, cross-reference against the project's stated license tier, and flag any GPL artifact that lacks the Classpath Exception (see the false-positive list below; this is the single most common misread in Java license posture). |
| Architecture | `jdeps` (JDK-bundled), `pf4j` / `archunit`, manual `import` and Spring `@ComponentScan` grep | `jdeps` ships with the JDK and produces a class-level dependency graph for an entire JAR or `target/classes/` tree. `ArchUnit` test rules, when the project has them, are themselves the architectural intent and worth reading for what the team considers in-bounds vs out-of-bounds. For Spring Boot specifically, the `@ComponentScan` base packages plus the `@Configuration` and `@EnableAutoConfiguration` chains define the runtime module surface and should be walked once end-to-end. |

## Spring Boot detection and framework signatures

Spring Boot presence changes the audit posture meaningfully (different default ports, different config-precedence rules, different deserialization surface, different SCA exposure via the `spring-boot-starter-*` umbrella deps). Detection signatures, in priority order:

1. A `spring-boot-starter-parent` parent POM or a `spring-boot-dependencies` BOM import in `pom.xml`.
2. The Spring Boot Gradle plugin (`id 'org.springframework.boot' version '...'`) in `build.gradle`.
3. A class annotated with `@SpringBootApplication` (almost always one per module, usually named `*Application.java`).
4. An `application.properties` or `application.yml` in `src/main/resources/` containing `spring.*` keys.
5. A `META-INF/spring.factories` or `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` file inside a JAR or in `src/main/resources/META-INF/`.

When any of the above are present, the packet calls out the Spring Boot major version (Spring Boot 3.x is the current line in 2026; Spring Boot 2.x is on community-only support past November 2023; Spring Boot 1.x is unsupported), the embedded servlet container (Tomcat, Jetty, or Undertow, picked by which `spring-boot-starter-web` variant is on the classpath), and any `spring-boot-starter-actuator` exposure since Actuator endpoints were a primary lateral-movement surface in several historical CVEs. Other gravitational frameworks to detect with the same rigor: Quarkus (`io.quarkus:quarkus-bom`), Micronaut (`io.micronaut.platform:micronaut-platform`), Jakarta EE (`jakarta.platform:jakarta.jakartaee-bom`, formerly Java EE), Apache Struts (legacy and security-sensitive; presence of `struts2-core` is itself a finding given the family's CVE history), and Android (`android.gradle.application` plugin, source under `src/main/java/` or `src/main/kotlin/` with an `AndroidManifest.xml`).

## Configuration and secrets patterns

Java secrets management is unusually fragmented across three eras and a packet covers all three:

* **`.properties` files** (`application.properties`, `bootstrap.properties`, custom `*.properties` files loaded by `PropertyPlaceholderConfigurer` or Spring Boot's `@PropertySource`). Keys are flat `key.subkey.value=secret`. The Spring convention is `spring.datasource.password`, `spring.security.oauth2.client.registration.*.client-secret`, and any `*.password` or `*.secret` or `*.api-key` substring. Gitleaks catches the literal-value cases; the harder cases are placeholder values like `${DB_PASSWORD}` paired with a `.env` or `application-local.properties` that did get committed.
* **`application.yml` (and per-profile variants like `application-prod.yml`, `application-staging.yml`)**. Same key surface as `.properties` but the YAML nesting changes the grep pattern. Spring Boot's profile-activation system means `application-prod.yml` may contain real production credentials while `application.yml` is sanitized; both files need a credential pass. The encrypted-properties pattern (Spring Cloud Config with `{cipher}` prefix, or Jasypt's `ENC()` wrapper) is safe only when the decryption key is genuinely outside the repo; a packet verifies that and does not assume.
* **Externalized secrets** (Vault, AWS Secrets Manager, Spring Cloud Config Server, environment variables). When externalization is in place, the packet names the system, names the principal under which the application authenticates to it, and confirms no fallback default values in source. The common failure mode is a hardcoded `default-pw-123` left in a `@Value("${db.password:default-pw-123}")` default that was meant to be temporary.

A clean Java credential packet includes a pass over `src/main/resources/**/*.properties`, `src/main/resources/**/*.yml`, `src/main/resources/**/*.yaml`, any `src/main/resources/META-INF/` directory, and the git history of all four locations (per the standard `secrets in git history` finding pattern). For Kotlin/JVM and Android, also `src/main/AndroidManifest.xml` (which can carry API keys in `<meta-data>` tags) and `local.properties` (Android Studio's project-local file, regularly excluded by `.gitignore` but regularly forgotten on first commit).

## Java LTS version EOL and runtime hireability

Java's LTS cadence is one major release every two years, with each LTS supported by Oracle for an 8-year free-tier window and by alternative builds (Eclipse Temurin / Adoptium, Amazon Corretto, Microsoft Build of OpenJDK, Azul Zulu) on overlapping cycles. The 2026 picture, anchored to public OpenJDK and Oracle support roadmaps:

| Version | Status | Notes |
|---|---|---|
| Java 8 | EOL for free Oracle JDK in March 2022; extended commercial support and Temurin/Corretto free LTS continue through 2030+ | Still appears in older enterprise codebases; presence is itself a finding for new audits because the upgrade work to 17 or 21 is non-trivial. |
| Java 11 | EOL April 2026 (free Oracle JDK premier support); Temurin, Corretto, and Azul continue through 2029-2032 | Common in 2018-2021-era codebases. EOL date is the active inflection point for buyer-side acquisitions in 2026; a project on Java 11 with no migration plan is a documented finding. |
| Java 17 | LTS, supported; free Oracle premier support through September 2026, extended through September 2029, alternative builds through 2032+ | The most common modern enterprise target; default for Spring Boot 3.x. |
| Java 21 | LTS, current; free Oracle premier support through September 2028, extended through September 2031 | The default for new projects in 2026. Virtual threads (Project Loom) and pattern matching shipped here. |
| Java 25 | LTS, current (released September 2025); free Oracle premier support through September 2030, extended through September 2033 | The newest LTS at audit time. Adoption is still climbing; presence of Java 25 in a new project is normal, presence in a five-year-old codebase that just upgraded is a positive signal. |
| Java 22, 23, 24 | Non-LTS, six-month cadence releases, EOL six months after each release | Should not appear as a production target. If a project pins one, the packet names it as a finding and recommends the nearest LTS. |

The packet states the Java version in use (read from `pom.xml` `<source>` and `<target>` or `<maven.compiler.release>`, or from `build.gradle` `sourceCompatibility` / `targetCompatibility` / `jvmTarget`), the JDK distribution that CI builds against (the GitHub Actions `setup-java` distribution field is the easiest source), and the production runtime JDK distribution if it can be confirmed. A divergence between the declared `<release>` target and the CI distribution is a finding worth surfacing; a divergence between CI and production is a worse one.

## Typical false positives

* OWASP `dependency-check` flags Spring Framework artifacts against historical CVEs (Spring4Shell, the Tomcat connector chains) on versions that were already patched but where the artifact filename uses a SemVer suffix the matcher does not parse. Read the actual remediated-version note in the CVE record, then suppress with a written rationale that names the exact patched version present.
* The Log4Shell finding (CVE-2021-44228) and its successor chain (CVE-2021-45046, CVE-2021-45105, CVE-2021-44832) still fires against `log4j-core` artifacts in transitive dep trees even when the parent project has explicitly pinned `log4j-core` to a remediated version (2.17.1 or later). The false positive comes from intermediate dep manifests that name an older range; the resolved tree is what matters. A clean Java packet runs `mvn dependency:tree` (or the Gradle equivalent), grep for `log4j-core`, and cite the resolved version in the suppression note. The historical-CVE family is also worth a once-over on `logback-classic` (CVE-2023-6378 and earlier) and on the Spring4Shell chain in `spring-beans` even when the headline finding is on Log4j; auditors trained on Log4Shell sometimes miss the adjacent ones.
* SpotBugs + find-sec-bugs flags any `ObjectInputStream.readObject()` call as deserialization risk. The pattern is genuinely dangerous and was the root of the Apache Commons Collections RCE chain that powered most of the 2015-2017 deserialization wave, but find-sec-bugs cannot tell whether the input stream is trusted (a JVM internal cache, a sealed cluster RPC) or untrusted (an HTTP body, a user upload). Confirm the data flow before promoting; suppress with a one-line rationale that names the input source.
* SpotBugs flags `new Random()` (versus `SecureRandom`) inside test code as cryptographic weakness. Suppress for the test source set only, not globally; the test code does not need cryptographic randomness and substituting `SecureRandom` slows down test suites measurably.
* JaCoCo coverage reports can show 100% on a Gradle multi-module build where the aggregation plugin is misconfigured and only one of three modules is actually being measured. Verify the `jacocoTestReport` task scopes by reading the build output, not just the percentage. The Maven equivalent is `jacoco-maven-plugin` with a misconfigured `<dataFile>` or `<dataFileIncludes>`.
* Generated Java sources (Lombok-generated getter/setter methods, MapStruct-generated mappers, Protobuf-generated stubs, gRPC stubs, JPA-generated metamodel classes ending in `_`) get scanned by SpotBugs and SCA and report findings that the developer did not write and cannot meaningfully fix. Exclude generated source directories (`target/generated-sources/`, `build/generated/`) from the SAST scope with a written note.
* Snyk's "vulnerable method" enrichment reports a finding as exploitable when the vulnerable class is on the classpath but the vulnerable method is never invoked by application code. Snyk's call-graph analysis covers common patterns but not every framework's reflection chains; verify by hand for findings on classes that are loaded only via Spring's `@Bean` factory methods or via SPI (`META-INF/services/`).
* Gitleaks flags Spring `application.properties` placeholder values like `spring.datasource.password=${DB_PASSWORD}` as credentials because the regex matches the key. The literal `${...}` interpolation is safe; promote only when the value is a concrete string.
* Test fixtures and integration-test resources (under `src/test/resources/`, `src/integrationTest/resources/`, or any `test-fixtures/` directory) often contain example API keys, JWT samples for token-parsing tests, and self-signed TLS keystores. Mark `is_sensitive: false` with a note pointing at the fixture path.

## License posture (Apache 2.0 vs GPL with Classpath Exception)

Java license posture is the place where audit writeups most often misread the legal exposure, because the JVM ecosystem has a license pattern that does not appear elsewhere: the GPL Classpath Exception.

The OpenJDK itself is GPL v2 + the Classpath Exception, which says (paraphrasing) that linking the GPL-licensed standard library into a proprietary application does not force the application to be GPL. This is what makes it legal to ship closed-source software on the OpenJDK runtime at all. The exception language is specific to "linking" and "module path / classpath inclusion"; it does not cover modifying the GPL-licensed code itself.

Where audits go wrong: an artifact in the dep tree is marked GPL v2 without the Classpath Exception annotation, and the auditor flags it as a viral-copyleft cascade that infects the whole application. In most cases the artifact does carry the Classpath Exception (the `LICENSE` file in the JAR will say so explicitly) but the SCA tool's metadata field reported only "GPL-2.0" and dropped the exception. A clean Java packet inspects the actual `LICENSE` and `NOTICE` files inside each flagged JAR (or reads the upstream repository) before promoting a GPL finding. The genuine cascade risk lives with:

* GPL v2 without Classpath Exception (rare; some legacy DB drivers, some legacy embedded libraries).
* AGPL v3 (always viral; check carefully whether the application is network-exposed in a way that triggers the AGPL distribution clause).
* GPL v3 without Classpath Exception (uncommon but exists).

Apache 2.0 is the dominant Java OSS license and is permissive; it requires attribution in a `NOTICE` file and a copy of the license text in distributions, both of which a packet verifies (the absence of a top-level `NOTICE` file when the project includes Apache-licensed dependencies is a common documented-but-low-severity finding). Eclipse Public License (EPL 1.0 and 2.0) appears on Jakarta EE artifacts and is weak copyleft; the cascade is bounded to modifications of the EPL'd code itself. MIT and BSD-2-Clause appear on smaller utilities; permissive, attribution-only.

## Minimum acceptable depth

* All ten tools above run cleanly with no skipped checks on the most recent commit. If the project pins an EOL or near-EOL JDK that the scanners refuse to run on, the packet names the Java version, the EOL date, and the upgrade work required before the scanners run. The Java 11 EOL date (April 2026) is the most common active inflection point for 2026 audits.
* OWASP `dependency-check` runs with the default `--enableExperimental` off (to avoid noise) but with the NVD data feed current as of the audit date (stale NVD data is a documented packet caveat). Snyk runs against the same commit. Results capped at 8 high, 12 medium, 6 low in the packet to avoid finding-flood.
* SpotBugs + find-sec-bugs runs with the full check set at `effort=max` and `threshold=low`; results capped to the same 8 / 12 / 6 envelope. Semgrep runs with `--config=auto` plus any Spring-specific or Jakarta-specific rule packs the project applies. Any suppressed check is named with a one-line rationale.
* Test coverage measured, not assumed: actual line and branch percentage from JaCoCo's XML output (`target/site/jacoco/jacoco.xml` or `build/reports/jacoco/test/jacocoTestReport.xml`), not a test-class-count heuristic. For Gradle multi-module projects the per-module percentages are reported separately and an aggregated coverage figure is computed only when the `jacocoTestReport` aggregation task is correctly configured (verify by reading the build script, not the dashboard).
* The third-party vendor inventory cross-references all artifacts declared in the resolved dependency tree (`mvn dependency:tree -Dscope=compile,runtime` plus `-Dscope=test` reported separately, or `gradle dependencies --configuration runtimeClasspath` plus the test configurations) against the vendor signature table. Manifest-only review is insufficient because Maven's nearest-wins and Gradle's highest-wins resolution rules change which version actually ships. Any vendor not in the table gets a manual lookup before the packet ships.
* Credential exposure check includes a deliberate pass over `src/main/resources/**/*.properties`, `src/main/resources/**/*.yml`, `src/main/resources/**/*.yaml`, `src/main/resources/META-INF/`, `src/test/resources/` (for accidentally-real test credentials), and the git history of all of the above. For Android modules, add `AndroidManifest.xml` and `local.properties`. The encrypted-properties patterns (Spring Cloud Config `{cipher}`, Jasypt `ENC()`) are safe only when the decryption key is verifiably outside the repo and the git history confirms it was never committed.
* License posture names the application license, the artifact licenses pulled in transitively, and any GPL or AGPL artifact that lacks the Classpath Exception. The output of `license-maven-plugin` or `license-gradle-plugin` is included as an appendix artifact, not summarized only. NOTICE-file presence is verified against the Apache-licensed dep list.
* Architecture findings cite specific file and line locations. For Spring Boot: `src/main/java/**/*Application.java` (the entry-point class), the `@ComponentScan` base-package surface, the `@Configuration` classes, the `@RestController` and `@Controller` controllers, and any `@EnableAutoConfiguration` exclusion list (which itself encodes architectural intent). For non-Spring Java: the `Main-Class` from `META-INF/MANIFEST.MF`, the JPMS `module-info.java` if present (Java 9+), and the package-level `package-info.java` annotations. Abstract diagrams without `file:line` citations do not ship.
* Stack hireability writeup names the Java version, the build tool, the framework (Spring Boot / Quarkus / Micronaut / plain JVM), and an honest 2026 read on the hiring market. Java hiring is mainstream-tier (per the methodology's stack classification), high supply across all major US metros, deeper at the senior end than JavaScript or Python, and concentrated in enterprise and bank-adjacent firms. Kotlin/JVM hiring tracks Android primarily and JVM-backend secondarily; supply is narrower than Java but improving, and a Java team can read Kotlin source after a short onboarding (the reverse is also true).
* Key-person risk callouts surface the patterns specific to Java codebases: heavy use of reflection or dynamic proxies (`java.lang.reflect.Proxy`, ByteBuddy, cglib) concentrated in one author's commit history, custom annotation processors (`javax.annotation.processing.Processor` or JDK 9+ `javax.annotation.processing.AbstractProcessor`) that generate code other developers must read at runtime, custom Spring `BeanPostProcessor` or `BeanFactoryPostProcessor` implementations that mutate the application context in ways that defeat IDE navigation, and JPA `@EntityListeners` chains that introduce side effects on save. These are the patterns that make a Spring Boot codebase readable to its original author and opaque to a new senior hire; they are also the patterns that AI specialists handle better than deterministic scanners.

### What deterministic tools cover, and what they do not

Deterministic Java tools (OWASP `dependency-check`, `spotbugs` + find-sec-bugs, `semgrep`, JaCoCo, `license-maven-plugin`) cover the SCA, credential-adjacent, SAST, coverage, and license zones cleanly. They are fast, reproducible, and citation-grade. Snyk adds commercial polish and reachability analysis on top.

What they do not cover well, and what an AI specialist pass adds: reading Spring `@Configuration` classes to detect whether bean overriding crosses module boundaries in a way that breaks framework upgrades, recognizing JPA entity inheritance patterns (`@Inheritance(strategy = JOINED)` vs `SINGLE_TABLE` vs `TABLE_PER_CLASS`) that lock the schema into long-term constraints, distinguishing a sound annotation-processor pattern (Lombok, MapStruct) from a project-internal annotation processor that lacks public documentation and acts as a key-person dependency, and surfacing the runtime-classloader assumptions in OSGi or Spring Boot fat-JAR layouts that break silently when a new dependency reorders the classpath. Those are the findings that a Closeread-grade Java packet adds on top of the deterministic baseline.
