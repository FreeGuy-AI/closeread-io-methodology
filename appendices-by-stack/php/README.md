# PHP appendix

Tools an auditor reaches for when the primary stack is PHP, the typical false positives to suppress, and the minimum acceptable depth of coverage for a Closeread-grade packet on a PHP codebase.

"PHP" in audit terms covers four shapes that show up in acquisition-side diligence, and the audit posture changes meaningfully across them. WordPress (and its plugin and theme ecosystem) is the overwhelming majority of public PHP code by line count, and the typical acquisition target if the seller calls themselves an agency or a publisher. Laravel is the modern application framework of choice for new SaaS builds and is the most common shape behind a "we built a custom backend in PHP" disclosure. Symfony shows up in enterprise and ecommerce contexts (Drupal and Shopware ride on Symfony components, Magento ships its own framework but borrows freely) and in any codebase whose authors came from a Java or .NET background. Plain-PHP legacy code, often pre-namespace, often pre-Composer, still exists in the wild and still gets acquired. The toolchain below covers all four; the framework-detection step at the top of the audit decides which subset runs deeply.

## Tools

| Zone | Tool | Notes |
|---|---|---|
| Framework detection | manifest inspection | Read `composer.json` for `require` entries naming `laravel/framework`, `symfony/*`, `drupal/core`, `wordpress/*`, or `roots/wordpress`. Read the repo root for `wp-config.php`, `artisan`, `bin/console`, or a `web/index.php` Drupal bootstrap. The framework decides which security and architecture tools below run as primary versus supplementary. |
| SCA | `composer audit`, `roave/security-advisories`, `osv-scanner` | `composer audit` (built into Composer 2.4+) checks `composer.lock` against the [FriendsOfPHP/security-advisories](https://github.com/FriendsOfPHP/security-advisories) database. Adding `roave/security-advisories` as a dev dependency causes Composer to refuse to resolve a tree that contains a known-vulnerable package, which is a structural defense rather than a scan. `osv-scanner` adds the OSV breadth that the PHP-only feed misses on transitive deps. Run all three. |
| Credentials | `gitleaks`, `trufflehog` | Same as Node, Python, and Ruby. PHP-specific patterns to watch: `.env` files paired with `vlucas/phpdotenv` in `composer.json` that shipped before `.env` made it into `.gitignore`, WordPress `wp-config.php` with literal database credentials and salts, Laravel `.env.example` versus `.env` discipline (the framework's own scaffolding makes the difference clear if you read both), and any `config/*.php` that returns an array containing literal API keys instead of `env('KEY')` calls. |
| Third party | manifest review + source grep | Parse `composer.json` and `composer.lock` against the vendor signature table. The lock file is authoritative for transitive resolution. For WordPress, also enumerate `wp-content/plugins/*/plugin-name.php` headers and `wp-content/themes/*/style.css` headers, because plugin and theme code is rarely managed through Composer and rarely shows up in any other manifest. |
| Test coverage | `phpunit`, `pest`, `infection` | PHPUnit is the canonical PHP test runner and the one that integrates with Xdebug or PCOV for coverage. Pest is a developer-experience layer on top of PHPUnit; if `tests/Pest.php` exists, the project uses Pest and the coverage flow is `vendor/bin/pest --coverage`. Either way, the coverage artifact lives at `coverage/` or wherever `phpunit.xml` points its `<coverage>` element. Infection runs mutation testing on top of the existing PHPUnit suite and is the right tool when the seller claims "high coverage" and the buyer wants to know whether that coverage actually catches bugs versus only exercising lines. |
| Security | `psalm`, `phpstan`, `phpcs`+`PHPCS_SecurityAudit`, `semgrep --config=auto` | Psalm and PHPStan are the two competing static analyzers and most modern PHP projects pick one; both surface security-relevant issues at higher levels (Psalm `--level=2` or stricter, PHPStan `--level=8`). `PHPCS_SecurityAudit` is the closest PHP analog to Brakeman for Rails: it runs as a PHPCS standard and flags PHP-specific sinks (`eval`, `system`, `unserialize` on untrusted input, SQL string concatenation, XSS in echoed output). For WordPress specifically, `wpscan` is the canonical vulnerability scanner against installed plugins and themes and is non-negotiable when `wp-config.php` is present. Semgrep covers polyglot patterns the PHP-only tools do not run. |
| License | `composer licenses`, `madewithlove/license-checker` | License metadata lives in each package's `composer.json`. `composer licenses` produces the report; the WordPress GPL cascade is the single most important license check on a PHP codebase and is covered in detail in the false-positive section below. |
| Architecture | `deptrac`, `phploc`, manual `namespace`/`use` grep | Deptrac enforces architectural layer boundaries declared in a `deptrac.yaml` and reports violations; for any audit where the seller claims a domain-driven or hexagonal layout, Deptrac's output is the verification. `phploc` gives the controller, model, and namespace line counts plus the cyclomatic complexity histogram. For Laravel and Symfony, the framework's own console (`php artisan route:list`, `bin/console debug:container`) surfaces the runtime wiring that static analysis cannot see. |

## Typical false positives

* `composer audit` and `roave/security-advisories` flag advisories against packages that the application only loads in `require-dev` (test runners, static analyzers, code generators). A clean packet names the package, names whether it is in `require` or `require-dev`, and explains why the production runtime does not pull the vulnerable code path. Suppress in the packet only with a written rationale. The Composer scopes are stricter than Bundler groups; `require-dev` packages are genuinely absent from `composer install --no-dev` deployments.
* Psalm and PHPStan flag `mixed` return types and dynamic property access in any codebase that predates PHP 7.4 strict typing or PHP 8.0 named arguments. These are real type-safety gaps but not security findings; promote them only when the dynamic path crosses a trust boundary. The packet's static-analysis section should report the analyzer's level, the suppressed-baseline count if a baseline file exists, and an honest read on whether the suppressions are hiding genuine debt.
* `PHPCS_SecurityAudit` flags `$_GET`, `$_POST`, and `$_REQUEST` access as unsanitized input even when the controller calls a Symfony `Request` object or Laravel `Request::validate` layer above it. Read the entrypoint end-to-end before promoting the finding; the framework router and middleware usually handle sanitization above the level the standard inspects.
* WordPress-specific: `wpscan` flags every installed plugin against its vulnerability database, including plugins that are inactive in `wp_options.active_plugins` but still present in `wp-content/plugins/`. Inactive plugins still execute if reactivated and are still in the audit perimeter, but the severity calculus changes if the seller can demonstrate they have been inactive across the entire audit window. Mark the finding, do not silently drop it.
* `infection` reports mutation score independently of line coverage; a project at 90% line coverage and 30% mutation score is a project whose tests exercise code without asserting behavior. This is a finding, not a false positive, and the packet names both numbers.
* `gitleaks` flags Laravel `APP_KEY` values in committed `.env.example` files when those values are placeholders or empty strings. Read the surrounding `.env.example` before promoting; a genuine leak is usually a base64-encoded 32-byte key committed as a literal in `.env`, not the empty `APP_KEY=` line in `.env.example`. The pattern that does ship leaks is `wp-config.php` with literal `AUTH_KEY`, `SECURE_AUTH_KEY`, and the other six WordPress salts as committed string literals.
* Test fixtures in `tests/fixtures/`, PHPUnit data providers, and Pest dataset files routinely contain example API keys and bearer tokens that look real to credential scanners. Mark `is_sensitive: false` with a note pointing at the fixture path.
* `FriendsOfPHP/security-advisories` lags upstream CVE feeds by days for less popular packages, similar to `ruby-advisory-db`. Cross-checking against `osv-scanner` is what keeps a PHP packet from missing a known issue that had not yet been merged into the PHP-only database. WordPress core vulnerabilities specifically are tracked in the WPScan database, not in the Composer security advisory feed, which is why the WordPress audit path requires `wpscan` as a separate pass.

## Minimum acceptable depth

* All eight tools above run cleanly with no skipped checks on the most recent commit. If the project pins an older PHP that `psalm`, `phpstan`, or `composer audit` no longer support, the packet names the PHP version, the EOL date (PHP 8.1 EOL November 2024, PHP 8.2 EOL December 2025, PHP 8.3 EOL December 2026, PHP 8.4 is the current supported release in 2026), and the upgrade work required before the scanners run. A PHP 7.x codebase in 2026 is itself a finding: all 7.x branches are long EOL, no security patches ship, and the upgrade path to 8.x is not a free recompile because of the strictness changes and the deprecation cleanup.
* Psalm or PHPStan runs at the project's declared level with no baseline suppression added during the audit. If a baseline file exists, the packet reports the suppressed-issue count, samples five, and gives an honest read on whether the suppressions represent real debt or are noise from an older codebase that has since been refactored. Results capped at 8 high, 12 medium, 6 low in the packet to avoid finding-flood.
* For WordPress: `wpscan` runs against the live `wp-config.php`-derived URL or against a local mirror, with API token configured so plugin and theme vulnerability data is current. Every active and inactive plugin is enumerated against the WPScan database. The packet names the WordPress core version, the active theme, and each plugin with its installed version and the highest unpatched CVE if any.
* Test coverage measured, not assumed: actual percentage from `phpunit --coverage-text` or `pest --coverage` with Xdebug or PCOV enabled, not a test-file-count heuristic. If the project advertises a coverage number in its README or CI badge, the packet reports the audited number alongside it; the two often disagree because the badge is stale. For codebases where the seller emphasizes test quality, the Infection mutation score gets reported alongside the line-coverage number; a strong gap between the two is a finding.
* The third-party vendor inventory cross-references all packages declared in `composer.lock` (not just `composer.json`, which omits transitive resolution) against the vendor signature table. Any vendor not in the table gets a manual lookup before the packet ships. Packages with no published license metadata in their `composer.json` get flagged separately and resolved by reading the upstream repo. For WordPress, the inventory also enumerates installed plugins and themes; these are not in `composer.json` for most installs and represent the larger third-party surface in practice.
* Credential exposure check includes a deliberate pass over `.env`, `.env.example`, `.env.local`, `wp-config.php`, `config/*.php` (Laravel), `app/config/*.php` (older Laravel and Symfony 2.x), `parameters.yml` and `parameters.yaml` (Symfony), `local.xml` (Magento 1) and `app/etc/env.php` (Magento 2), and any `*.sql` or `*.sql.gz` dump file in repository history. The `vlucas/phpdotenv` pattern is safe only when `.env` is in `.gitignore` and never committed; a single historical commit of `.env` is a finding even if the current `HEAD` is clean. The `wp-config.php` pattern is structurally different: the file is required at the application root for WordPress to boot, so the safe variant uses `getenv()` or a sibling `wp-config-local.php` that is gitignored. A `wp-config.php` with literal credentials in the committed copy is the modal WordPress finding and ships as Critical when paired with a public repository.
* License posture for WordPress codebases gets a dedicated zone in the packet: WordPress core is GPL-2.0-or-later, and the GPL cascade applies to any plugin or theme that derives from WordPress APIs in a way that creates a derivative work. The interpretation is contested in detail but uncontested in outline: a paid premium plugin that hooks into `add_action` and `add_filter` is treated by the WordPress Foundation, by most major plugin marketplaces, and by the courts that have ruled on adjacent cases as GPL-bound. The packet names every plugin and theme whose license declaration conflicts with GPL inheritance, names whether the conflict is theoretical (commercial license stated, no enforcement attempt) or active (DRM, license-key check, "premium" code paths that error if a key is missing), and gives the buyer an honest read on whether the IP they are acquiring is encumbered. This is the single most under-discussed risk in WordPress acquisitions and the one that surprises buyers post-close most often.
* License posture for Laravel and Symfony projects names the application license, the package licenses pulled in transitively, and any GPL or AGPL package that would force the application to inherit a copyleft obligation. The output of `composer licenses` is included as an appendix artifact, not summarized only. AGPL is the higher-stakes case for SaaS acquirers; GPL alone does not trigger source-disclosure obligations for hosted software that is not distributed.
* Architecture findings cite specific file and line locations. For Laravel: `app/Http/Controllers/`, `app/Models/`, `app/Services/` (or `app/Actions/`, `app/UseCases/`, whichever service-object convention the project uses), and any `app/Http/Middleware/` that exceeds 200 lines (a common monolith decomposition smell). For Symfony: the bundle layout under `src/`, the service container definitions under `config/services.yaml`, and any controller that injects more than five services (a hexagonal-architecture violation that usually predicts upcoming refactor cost). For WordPress: the active theme's `functions.php` size, the plugin count, and any custom plugin that bypasses the WordPress hooks system and writes directly to the database. Abstract diagrams without `file:line` citations do not ship.
* Stack hireability writeup names the PHP version, the framework version (Laravel 11 is the current release line in 2026; Laravel 10 is the most recently supported LTS through February 2026; Symfony 7.1 is the current minor and Symfony 6.4 LTS is supported through November 2027), and an honest 2026 read on the hiring market. PHP and Laravel hiring is deep and global, broader than Ruby or Go, narrower per senior candidate than Node or Python, and concentrated in firms that already run PHP at scale. WordPress hiring is the deepest pool of any web stack on the planet, but the senior-PHP-with-WordPress-architecture-experience subset is meaningfully narrower than the raw "WordPress developer" pool would suggest.
* Key-person risk callouts surface the patterns specific to PHP codebases: heavy use of magic methods (`__call`, `__get`, `__set`, `__invoke`) concentrated in one author's commit history, custom service container bindings in Laravel that are not documented anywhere outside the author's head, Symfony compiler passes in `src/DependencyInjection/Compiler/` that rewrite the container at compile time, and any WordPress plugin or theme that monkey-patches a vendored library or uses `remove_action` to disable framework hooks in ways that depend on load order. These are the patterns that make a PHP codebase readable to its original author and opaque to a new senior hire; they are also the patterns that AI specialists handle better than deterministic scanners.

### What deterministic tools cover, and what they do not

Deterministic PHP tools (`composer audit`, `roave/security-advisories`, `psalm`, `phpstan`, `PHPCS_SecurityAudit`, `phpunit`, `composer licenses`, `wpscan` for WordPress) cover the SCA, credential, SAST, coverage, and license zones cleanly. They are fast, reproducible, and citation-grade.

What they do not cover well, and what an AI specialist pass adds: reading Laravel service providers to detect whether they register state that leaks across HTTP requests under PHP-FPM, recognizing Symfony event listener registration patterns that are correct in isolation but cause double-firing when the same event is dispatched from both a console command and an HTTP request, distinguishing a sound WordPress hook composition from a fragile one that depends on the order plugins load in, and surfacing the architectural drift that happens when a Laravel monolith has been decomposed into `app/Services/` over several years by different hands. Those are the findings that a Closeread-grade PHP packet adds on top of the deterministic baseline.
